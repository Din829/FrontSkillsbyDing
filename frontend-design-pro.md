用中文交流。本 skill 是一套**功能型界面**的设计方法论（不是营销落地页），自成体系，从问对问题到 Pencil 实操全程可操作。

核心信条：**站在用户视角，给用户的永远是简单、易懂、清晰、大方的界面。复杂留给后台，不丢给用户。**

第二信条：**UI 不撒谎。** 后端给不了的数据，界面不要假装有——拿不到就如实显示"—"/留空/说不知道，绝不编一个看起来合理的视觉糊弄用户。宁可朴素诚实，不要精致欺骗。

第三信条：**好看 ≠ 好网站，价值排序别搞反。** 动手前先确认两件比美观更靠前的事：①**主体是谁/给谁用**——政府办事网站再朴素也有人必须用，受众和场景先定，决定了该克制还是该有冲击力；②**内容说清了没有**——这网站是干嘛的、提供什么、怎么用，第一眼讲明白了吗。这两件立住，**再谈好看**。好看与强冲击力确实重要（尤其营销/品牌站），但它是"主体+内容"立住之后的放大器，不是替代品——本末倒置就是"精致的没用"。

## 一、开工前先问对问题（最容易跳过、代价最大）

别急着画。先锁定四件事，缺一个就停下来问，不要猜：

1. **给谁用？** 终端用户的专业程度决定术语能露多少。
2. **什么气质（Tone）？** 极简 / 工业 / 编辑 / 温润 / 活泼……必须先定一个明确主张。"刻意"比"强烈"重要。
3. **文案语言？** 语言影响字体选择（见下）。
4. **目标技术栈能还原吗？** 早问。别画完精致稿才发现选的框架做不出来（见第七节）。

> 经验：跳过这步，往往要推翻重画两三版才找对调子。先问，省后面。

## 二、定调：参考要消化，不要照搬

- 找 2-3 个参考，提炼"它为什么好"，而不是抄表面。**直接照搬某个知名产品的配色/布局 = 同质化**，没有自己的主张。
  - **找参考的现成弹药**：`参考资料/设计系统库/` 有 73 个大牌的 DESIGN.md（Apple/Linear/Stripe/Ferrari…），翻气质接近的当灵感——**消化"它为什么好"，别照搬色值**。没有合适的再用 `克隆skills-OneClickClone/` 截图/扒真站。
- 主动识别**反面教材**：蓝紫渐变大横幅、堆引流按钮、通用系统字体——看到就避开。
- 定调后用一句话能说清"什么让人记住"。说不清，就是还没定好。**自检：截图抹掉 logo，别人还能认出是它吗？** 功能型界面靠的是信息结构和排版节奏的辨识度，不是靠视觉噪点。

## 三、美学准则

> 以下整合自 Anthropic 官方 `frontend-design` skill（原则部分几乎一致，作为美学基座）。
> **重要立场**：官方原则面向创意/营销页，鼓励"大胆、氛围、打破网格"。本指南主场景是**功能型界面**——取其"选好字体、配色有明确主张、避 AI 套路"，但"打破网格/重氛围/强动效"要**克制取用**，永远服从第五节的"简洁清晰大方"。两者冲突时，以简洁为准。
> **同理**：社区 `design-spells`（微交互/"魔法"细节库）要求"对每个组件都注入魔法"——这是营销/展示页的调味，功能型界面克制取用：只在不加重认知负担、不干扰读数与操作时才点缀。一个让用户分心或卡顿的"魔法"，比没有更糟。

开工前先确立一个**明确**的美学方向，再落地。关键是**有意图、执行精准**，而非强度——功能型界面的"意图"就是简洁清晰，不是视觉冲击。下面每条都保留官方"选好字体、配色有主张、避 AI 套路"的内核，但取舍标准统一换成功能型语境。

- **字体（Typography）**：选有个性、好看的字体，提升质感。**避免通用字体**（Inter / Roboto / Arial / 系统默认）。用一个有特点的显示字体配一个精致的正文字体即可——功能型界面字体是"有质感"而非"抢戏"，别堆三四种。
  - 多语言注意：字体要含目标语言字形。例：日文要高级衬线感必须用 **Noto Serif JP（明朝）**，不能套西文衬线（西文衬线遇日文会回退成系统字体，四不像）。
- **配色（Color）**：一个坚定、统一的主张。用 **CSS 变量**保持一致。**主色 + 锐利点缀** 胜过平均分布的胆怯配色。可参考 60-30-10（基调 60% / 主色 30% / 强调 10%）——功能型界面里"强调 10%"要更省，彩色优先留给状态/风险（呼应第四节）。
- **动效（Motion）**：把力气集中在**一次编排好的页面载入**（错位渐显）胜过零散微交互。HTML 优先纯 CSS，React 用 Motion 库。功能型界面动效服务于"看清状态变化"，不为炫技；数据/操作密集的界面宁可更克制。**真做动效就守住底线**：只动 transform/opacity（不触发重排）、honor `prefers-reduced-motion`、保证 60fps 不抖；做不到平滑就别做。
  - **要写具体实现**（滚动驱动、载入编排、React 动画、伪水平滚动）→ `frontend-motion-gsap.md`（GSAP 实操简约版，含 matchMedia 怎么 honor reduced-motion、useGSAP 怎么自动清理、为什么只动 transform/opacity）；要某插件/API 的完整配置 → `参考资料/gsap-skills/`（官方完整原文）。GSAP 主场是营销页/滚动叙事，功能型界面克制取用。
- **空间（Spatial）**：可以有想法的布局、留白和层次——但功能型界面**优先信息清晰**，不为"打破网格/不对称/重叠"而打破。这些手法官方营销页爱用，这里只在真能提升可读性时才用，否则老实对齐。
- **背景与质感**：别只铺纯色，可营造氛围和层次（渐变网格、噪点、几何纹理、戏剧化阴影、grain 叠加）——**但功能型界面要克制**，氛围服务于专业感，不喧宾夺主；数据型界面尤其别让背景质感干扰读数。

**绝不要**：通用 AI 套路字体、烂大街配色（尤其**白底紫渐变**）、可预测的千篇一律布局、缺乏语境特征的模板感设计。**每次设计都该不同**，别每次都收敛到同一个"安全选择"。

**复杂度匹配愿景**：极繁要繁得彻底（大量动效细节），极简要简得精准（间距、字体、细节的克制）。优雅来自把愿景执行到位。

## 四、反 AI 味清单（功能型界面尤其要查）

这几样是"AI 生成感"最重的来源，看到就改：

1. **彩色软底胶囊**（绿底 Ready、黄底警告、蓝底标签这种 pill）——状态尽量用文字/灰阶表达，**唯一的彩色留给唯一需要警觉的事**（如风险/错误），正常态留白。
2. **框中框**——别把每个元素都包进一个带边框/底色的盒子。容器只在有真实结构或功能时才用。靠**留白和极淡灰阶**分区，不靠边线。
3. **冗余小灰字 / 括号注释**——标题下的解释长句、`（refine）` 这种暴露内部名的括号，大幅删减。只在用户真会困惑处留一句极短的。
4. **抢戏的大数字/大标题**——别把次要信息（如成本、内部状态）做成视觉焦点。关键是有意图，不是用力。

## 五、用户视角：简洁、清晰、大方（核心信条）

- **复杂的、晦涩难懂的 UI/UX 不要，或藏起来。** 给用户的永远简单清晰。
- **内部术语必须翻译成人话或直接藏掉**：
  - 能用合理默认的（如模型 fallback、检索 mode、build 模式）→ **藏**，后台自动，用户不碰。
  - 必须露的 → **翻译成人话**（如 `strict`→"必ず確認/总是确认"，`naive/hybrid` 这类→藏或换成"关联检索"之类用户能懂的描述）。
- **功能全展示，但参数分层**（这是"简洁"和"不丢功能"的平衡点）：
  - 高频参数 → 主界面就地调（可折叠面板，不跳页）。
  - 低频全局配置（API key、阈值、模型）→ 独立设置页/区。
  - 专业旋钮 → 藏进默认，或设置页深处给懂的人。
- **设计稿要全（不漏功能），界面要简（不堆给用户）**——这两件事不矛盾：核对所有功能后，区分"用户要操作/感知的"（露）和"后台自动的"（不上界面）。
- **用真实数据画，以后端真实契约为准**：表格、字段、下拉选项、默认值都照真实接口/API 的返回画，不臆造列名/字段/选项。落地时尤其注意：**前端用到的字段名、默认值、可选项必须对照后端源码核对**——设计稿可能与后端脱节（画了后端不返回的字段、标错了"默认值"），**两者冲突时信源码，不跟设计稿的错标**。
- **别让用户像在烂软件里那样抓瞎**（站在"找不到、看不懂、被默认坑"的用户那边查三件事）：
  - **一个意图一个权威入口**：同一件事别拆成多个能互相覆盖的开关，否则用户调了 A 被 B 冲掉、还不知道为什么。（典型反例：某播放器"启动窗口大小"和"播放时窗口大小"是两个分散的设置，互相打架，用户怎么调都不对。）真要分场景，就在一处集中表达，或讲清优先级。
  - **默认值要对、开箱即用**：默认值选最符合直觉的那个，用户不碰设置也能正常用；别把反直觉的行为设成默认（反例同上：默认"按视频分辨率自动缩放窗口"反人类）。
  - **该露的要找得到**：分层（高频露/低频设置页/专业藏）之后，常用项命名说人话、放在用户预期的位置，别藏进反直觉的深处让人翻半天。

## 六、Pencil（.pen）实操手册

Pencil 是 .pen 设计文件的 MCP 编辑器（文件加密，只能通过 Pencil 工具读写，不能用普通文件工具）。**前提：MCP 已配置好，且目标 .pen 文件已在 Pencil 编辑器中打开。**

**标准工作流：**
1. `get_editor_state({include_schema: true})` —— 任何读写前必须先拿 schema（除非本会话已加载）。文件需先在编辑器打开。**若此调用因文件未打开而失败，直接告诉用户"请先在 Pencil 编辑器里打开目标 .pen 文件"，然后停下等用户操作——别自己改路径瞎试或换别的工具读。**
2. `get_guidelines()` 列出可用 guide/style；按任务类型加载相关 guide（如 "Web App"、"Table"）和一个 style 找视觉灵感。**别加载互不兼容的多个 guide。**
3. **变量驱动**：先用 `set_variables` 定义颜色/字体/间距变量，节点里用 `"$变量名"` 引用。改一处全局生效——换配色、换字体只动变量。
4. **逐屏建、逐屏验**：用 `batch_design`（JS 片段，支持循环/spread 减少重复）建一屏，**立刻 `get_screenshot` 截图核对**，不要全做完才看。新建/修改中的根 frame 加 `placeholder: true`，完成后移除。
5. 重复 UI（按钮/行/卡片）优先做成 `reusable` 组件再实例化。

**Pencil 常见塌陷/报错（实测踩过）：**
- **fit_content + fill_container 循环**：父 `fit_content` 同轴向有子 `fill_container` → 两者塌陷为 0。给父固定尺寸，或子别用 fill_container。
- **空容器塌陷**：frame 没有子节点且用 fit_content → 尺寸塌成 0 不可见。给固定尺寸或加子节点。
- **不支持 `alignItems: "baseline"`**（只有 start/center/end）；不支持 margin、百分比尺寸（用 fixed 或 fill_container/fit_content）。
- **图标名要核对**：lucide 等图标集名称写错会报 not found，按报错改名。
- **Replace 后残留**：替换节点后旧容器可能短暂塌陷报警告，截图确认实际结果即可。
- **文字必须设 `fill`**（否则不可见）；要换行必须设 `textGrowth: "fixed-width"` + width。
- **batch_design 跨调用不共享局部变量**：要跨调用引用的节点用 `myNode = Insert(...)`（不加 const/let）。

**截图节制**：截最小有意义的节点（一个 section 而非整个 document），大节点费 token 且看不清细节；结构/尺寸检查用 `snapshot_layout`，视觉保真才用 `get_screenshot`。

**设计稿 → 真实代码落地的通用坑（不限 Pencil，任何"照稿写前端"都适用）：**
- **图标库替换 DOM 节点 → 监听器/状态失效**：很多图标库（lucide 等）把占位元素（如 `<i data-icon>`）**替换成新的 `<svg>` 节点**。后果：① 之前绑在占位元素上的事件监听器随旧节点被丢弃 → 按钮点不动；② 之后再改那个已渲染节点的图标属性，库不一定重渲染 → 图标卡住。**对策：事件用委托或绑在不被替换的父节点上；图标的状态变化（如展开箭头翻转）用 CSS（rotate/显隐）而非改图标名。** 凡"会重建/替换节点"的库都有此类问题，不只图标库。
- **flex 垂直居中遇内容溢出会裁两头**：`justify-content:center` 在内容高于容器时把内容上下挤、顶部和底部都被裁、滚动也救不全。且 `min-height:100%` 在无确定父高时让 `flex:1` 子项塌陷。**对策：用 `100vh` 等确定高度，或内容区顶部对齐 + 整体滚动；居中只用于"内容必然短于视口"的场景。**
- **别把"展示性简化"照搬成死元素**：设计稿常把一个**本该可交互**的东西（输入框、可拖拽区）画成静态文字/装饰来示意。照搬就做成了点不动的死元素。**对策：落地前对照功能/后端能力判断——这是"展示示意"还是"用户真要操作的入口"？是入口就做成真控件。**

## 七、技术落地校准（别画了做不出）

- 设计稿是**北极星/目标**，但要早认清目标技术栈的还原边界。
- **遇到具体框架的还原边界，先查再测，别凭记忆下结论**：框架的能力随版本变，写死的"某框架只能还原几成"这种经验值会过时。正确做法是**深度搜该框架官方文档**（看它对自定义样式/组件/布局的支持边界）+ **写最小 demo 真跑截图**实测，拿到证据再判断。
  - 一般规律（仅作起点，仍需实测）：组件框架（如 Gradio）通常能套配色/字体，但自定义表格、弹层、精确间距较难还原；要像素级精致往往得手写 HTML/CSS（如 pywebview / Tauri / React）。具体到几分，以官方文档 + 实测为准。
- 设计与落地解耦：界面层可换，业务逻辑（适配层）不动——为"以后换前端壳"留好路。
- **区分"能静态验"和"必须真机验"的交互**：桌面 webview / 原生壳类方案，有些交互（拿本地文件真实路径、系统文件对话框、原生拖放、跨进程/线程的进度回调）**无头或浏览器里测不出来，只能在真窗/真机点一遍确认**。落地时心里要有这条线：能在浏览器/无头先验的尽量先验，剩下的明确标记"待真机验证"，别默认"代码写完=能用"。
- **离线 / 数据不出本机的场景，外部依赖要本地化**：开发期可以用 CDN 拉字体/图标/JS 库图省事，但若产物要离线运行或承诺"数据不出本机"，打包时必须把这些依赖**改为本地内置**（否则首次运行联网失败 = 字体回退、图标空白、功能挂）。这条容易拖到最后才发现，早规划。
- **改造 / 复刻已有产品时，守住"保留红线"——这几样绝不静默改**（接第五节"对照后端源码"：视觉可换，但下面这些一动就连带打断后端/数据/用户习惯，必须先得到明确同意）：
  - **表单字段名、顺序、可选值** —— 改了断 analytics 埋点、断浏览器自动填充、跟后端契约脱节。
  - **路由 / URL slug / 锚点 ID** —— 改了断书签、断站内跳转，营销站尤其断 SEO 排名。
  - **导航标签、按钮文案的"意图标识"** —— 下游埋点常按文案匹配，重命名 = 数据断点。
  - **品牌 logo / wordmark、法律 & cookie & 同意文案** —— 合规和品牌资产，不是设计自由发挥的地方。
  - **既有的可访问性 / 焦点状态 / alt 文本** —— 别为了好看回退掉无障碍。
  > 落地心法：**改造不是重写**。先 audit 出"哪些在做真实的事（埋点/SEO/契约/合规）"，把它们围起来，只在围栏外做视觉升级。（消化自 taste-skill 第11节 redesign 协议，裁掉营销站专属项，留普适红线。）
